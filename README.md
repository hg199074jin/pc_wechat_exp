## 1. 功能总览

WeChat EXP 是一款 Windows 平台下的微信聊天记录**备份、查看与分析**工具。下载后双击即可使用，无需安装 Python 环境。

启动后自动在浏览器打开 Web 操作界面（`http://127.0.0.1:5051`），仪表盘分为四大模块、共 12 个功能入口：

| 模块 | 功能 | 说明 |
|------|------|------|
| **数据采集** | 一键备份 | 自动备份微信聊天记录（含图片、视频、文件、语音），支持增量硬链接 |
| **浏览与分析** | 聊天查看器 | 像微信一样浏览聊天记录，支持搜索、筛选、媒体播放、语音转文字 |
| | 🤖 **AI 群聊分析** | 接入大模型，自动产出群聊每日总结报告（Markdown）+ 结构化分析产物 |
| | 🏷️ **群管理** | 群聊标签分类树，支持 AI 一键自动分类、手动调整、黑名单 |
| | 📋 通讯录 | 浏览完整微信通讯录，按姓名/备注搜索 |
| | 📡 **知识沉淀雷达** | 从群聊中自动发现高价值知识，生成可复用卡片，支持溯源与转化 |
| | 🔗 **同步到 Obsidian** | 把知识卡片导出成一卡一文件的 Markdown 笔记库，供 Obsidian / Claude Code 读取 |
| **导出** | 聊天导出 | 按联系人、时间范围导出为 TXT / HTML |
| | 员工报表 | 上传员工 Excel，批量匹配并导出聊天记录 |
| | 综合报告 | 生成 HTML 综合数据报告（概览卡片 + 饼图 + 趋势 + 热力图） |
| **工具** | 垃圾清理 | 按聊天/空间占用批量删除源微信数据，释放磁盘 |
| | 提取密钥 | 从微信进程内存提取数据库解密密钥 |
| | 解密数据库 | 使用已有密钥批量解密微信聊天数据库 |

> **相比旧版的变化**：年度报告、词云分析功能已下线；新增了 AI 群聊分析、群管理、知识沉淀雷达、同步到 Obsidian 等 AI / 知识能力。

## 2. 快速开始

**只需两步，即可开始使用：**

```
第一步：登录微信（建议刚登录后立即操作）
第二步：双击 wechat_exp.exe 启动程序 → 在 Web 仪表盘点击「一键备份」
```

启动后程序会自动在浏览器中打开 Web 操作界面（`http://127.0.0.1:5051`）。如果没有自动打开，请手动访问该地址。

> **提示**：首次使用时，微信必须正在运行且已登录，程序才能自动提取密钥并完成备份。之后的日常查看无需微信运行。
>
> ⚠️ **关键**：微信登录后，各数据库密钥仅在**登录初期一小段时间内**全部驻留在内存中。时间越久，微信越可能把久未访问的老数据库密钥从内存中清除。**建议在微信登录后尽快运行备份**。

**AI 功能需额外一步**：首次使用 AI 分析 / 知识雷达前，在「AI 群聊分析」页面的「⚙ LLM 设置」中配置一个 OpenAI 兼容大模型（详见 [第 6 章](#6-ai-功能配置详解)）。

## 4. Web 界面功能详解

### 4.1 数据采集 — 一键备份

这是您首次使用时的**第一步操作**。

**操作步骤：**

1. **重新登录微信**（建议：退出微信 → 重新登录，确保所有数据库密钥都在内存中）
2. 仪表盘 →「**数据采集**」→「**一键备份**」
3. 进入备份页面后，可选择备份的日期范围（默认最近 30 天）
4. 点击「**开始备份**」
5. 页面底部出现进度条，实时显示进度
6. 等待四个阶段依次完成

**备份过程分四个阶段：**

| 阶段 | 说明 | 大约耗时 |
|------|------|---------|
| ① 扫描 | 自动查找微信数据目录 | 几秒 |
| ② 解密 | 解密微信加密的数据库，**缺密钥时自动从微信进程提取** | 10–60 秒 |
| ③ 迁移 | 复制图片、视频、语音、文件 | 1–5 分钟 |
| ④ 索引 | 建立聊天记录统一索引，支持跨分片聚合 | 10–30 秒 |

> ⚠️ **重要**：如果部分数据库缺少密钥，程序会自动从微信进程内存中提取并重试（需微信正在运行）。**自动提取失败时，退出微信 → 重新登录 → 立即运行备份**通常即可解决。

备份结果保存在 `wechat_exp.exe` 所在目录下的 `backup\` 文件夹中。

> **增量备份**：命令行可用 `--link-dest` 指向前一次备份目录，媒体文件优先以硬链接方式复用，仅复制新增数据，大幅节省空间与时间。

### 4.2 聊天查看器

点击「**聊天查看器**」进入聊天浏览界面。

- **左侧**联系人列表：显示所有聊天对象（联系人、群聊），按最新消息时间排序
- **右侧**消息显示区：点击任意联系人/群聊查看对应记录

**头像系统（三级加载）：**
1. 从微信 CDN 代理加载真实头像，成功后缓存到 `output/avatar_cache/`
2. 群聊生成 2×2 成员首字符组合头像
3. CDN 不可达时生成 SVG 默认头像（基于 MD5 的确定性渐变色 + 首字符）

**名称解析（五级策略）：**
- **个人联系人**：备注 > 昵称 > 别名 > 微信号，智能跳过与微信号相同的无效字段
- **群聊名称**：contact.db 备注/昵称 → session.db 摘要 → 群成员昵称拼接 → 群主信息 → 降级方案

详细操作见 [第 5 章](#5-聊天查看器使用详解)。

### 4.3 AI 群聊分析（新）

接入大模型，对群聊进行每日结构化分析，产出可读的 Markdown 总结报告 + 机器可解析的分析产物（artifact）。

#### 配置 LLM（首次必做）

进入「AI 群聊分析」页面，点击左上角「**⚙ LLM 设置**」，填写：

| 配置项 | 说明 |
|--------|------|
| Base URL | OpenAI 兼容 API 地址，如 `https://api.openai.com/v1` |
| API Key | 你的 API Key（界面显示时会掩码，仅显示首尾 4 位） |
| Model | 模型名，如 `gpt-4o`、`deepseek-chat` 等 |
| Temperature | 0–2，默认 0.3（分析类任务建议偏低） |
| Max Tokens | 64–32000，默认 4096 |
| Timeout（秒） | 15–600，默认 120 |

填写后点「**测试连接**」验证，通过后「**保存**」。

> 配置存储在程序目录下的 `.wechat_exp_config.json`（参见 [第 10 章](#10-附录数据落盘位置汇总)）。
>
> **代理环境**：若你的网络需要 SOCKS 代理访问 LLM，程序已内置 PySocks 支持，设置系统 `HTTPS_PROXY` / `ALL_PROXY` 环境变量即可生效。

#### 分析工作流

1. **📋 选择群聊**：按群标签树分类展示（标签来源「群管理」页），可搜索、全选当前标签、勾选单个群
2. **📅 日期范围**：选择起止日期，有「最近 1/3/7 天」快捷按钮
3. **🔘 分析方式**（重要）：选完日期后，选择如何切分这一时段
   - **按时间段分析（默认）**：把整个起止区间作为**一份**报告。文件名以区间标签命名，如 `2026-06-01_to_2026-06-19.md`。适合做周报 / 月报式的阶段总结。
   - **按天分析**：区间内**每一天各产出一份**独立报告，文件名为单日 `2026-06-15.md`。适合逐日归档。
4. **▶ 开始分析**：后台逐群处理，界面以 SSE 实时显示进度（`分析中 → 提取产物 → 证据校验 → 渲染报告 → 完成`）

> 同一时刻只允许运行一个分析任务，避免并发冲击 LLM 服务。

#### 每次分析会产出两份产物

保存在 `<备份根目录>/ai_analysis/<群ID>/`，文件名取决于上面的"分析方式"：

- **`<日期标签>.md`** — 人类可读的 Markdown 总结报告（总体摘要 / 关键话题 / 待跟进事项）
  - 按天分析：`2026-06-15.md`
  - 按时间段分析：`2026-06-01_to_2026-06-19.md`
- **`<日期标签>.artifact.json`** — 结构化分析产物（话题、发言人、证据引用、知识候选），与报告同名同目录

> ⚠️ **注意**：artifact 里的 `knowledge_candidates`（知识候选）此时还**没有进入知识库**，它只是 AI"认为值得沉淀"的候选。真正入库沉淀发生在「知识沉淀雷达」，详见 [4.5 节](#45-知识沉淀雷达新)。

#### 报告里的"证据通过率"是什么

为保证 AI 引用的原文确实来自聊天记录（防止编造），程序会对每条证据做**确定性软校验**：

- 引用的 `msg_id` 是否真实存在
- 发言人是否与原始消息一致
- 引用文字是否逐字出现在原始消息中（忽略空白）

校验结果以「证据通过率 X%」呈现在结果卡片上，并在每条证据上标记 `✓ 已验证 / ✗ 未验证`。**未通过的证据不会被删除**，仅作标记，由你判断是否采信。

#### 超长聊天记录如何处理

当一次分析的可分析文本超过阈值（约 12000 字符）时，程序会：
1. 按行切分为多块（每块约 6000 字符，最多 8 块）
2. 每块单独调用 LLM 产出局部分析
3. 再调用 LLM 把多块合并去重为一份完整分析

> 按"时间段分析"跨度较大时（如整月），单群消息量可能触发分块，属正常现象。

#### LLM 输出不是合法 JSON 怎么办

分析产物要求 LLM 严格输出 JSON。若模型偶发输出格式错误（多余逗号、截断、Markdown 包裹等），程序会：

1. 先做容错提取（智能定位首个 `{` 到合法 JSON 结尾，忽略尾部冗余）
2. 若仍解析失败，**自动让同一 LLM 修复一次**（temperature=0，只修格式、不改语义）
3. 修复后仍失败才报错"LLM 输出不是合法 JSON"

这样能显著降低弱模型 / 长输出场景下的失败率。

#### 风格指纹（群级长期记忆）

为提升每个群的分析质量，程序会按群存储**群级风格指纹**（群定位、常见主题、内部黑话、输出偏好、禁忌），并在后续分析时注入 prompt。

> 隐私设计：**只存群级信息，不存任何个人画像或长期个人标签**。

#### 结果卡片的三栏内容

每个分析结果含三个 Tab：
- **分析报告**：渲染后的 Markdown
- **知识候选**：AI 识别出的、值得长期沉淀的知识条目（标题/类型/评分/正文）
- **证据来源**：每条引用的原始消息（含校验状态）

#### 把知识候选沉淀到知识库

点击结果中的「**沉淀高分知识**」按钮，可把评分 ≥ 80 的候选一键导入「知识沉淀雷达」（详见 [4.5](#45-知识沉淀雷达新)）。

#### 定时任务

支持新建定时分析任务（名称 + 每日执行时间 + 启用开关 + 选定群），由后台调度器自动每日分析**前一天**的数据。

### 4.4 群管理（新）

对群聊进行标签分类管理，作为 AI 分析和知识雷达"按群筛选"的基础。

#### 群标签树

- 多级树形分类，每个节点可挂多个群
- 操作：新建一级标签、右键（重命名 / 新建子标签 / 删除）、把群在节点间拖拽移动
- 「**未分类**」= 尚未打标签的群
- 删除某标签时，其下群聊自动回归未分类
- 所有修改自动保存（防抖 500ms）

#### AI 自动分类

点击「**🤖 AI 自动分类**」，程序会把全部群名分批送 LLM，要求自动归类（最多 15 个大类，不确定的归"其他"）。完成后弹确认框，你可选择是否应用。

#### 群黑名单

把不想分析的群加入黑名单，它们会从群管理、AI 分析、知识雷达的所有群加载中**全局剔除**。

- 在「未分类」下勾选群 →「**🚫 加入黑名单**」
- 「**🚫 黑名单管理**」可查看 / 移出

#### 加载范围与排序

为降低无价值旧群的干扰，加载群列表时可选择「最近 1/3/7/30 天有会话 / 全部群」，并支持按名称、最近对话、近 3 天活跃度排序。首次加载群名需从 contact.db 解码，以 SSE 流式带进度条展示，结果缓存 7 天。

### 4.5 知识沉淀雷达（新）

从群聊中自动发现**高价值、可复用**的知识，沉淀为结构化卡片，支持溯源、转化与导出。知识库存储在 `<备份根目录>/ai_analysis/knowledge.db`（SQLite）。

#### 先搞清：AI 群聊分析 vs 知识沉淀雷达的区别

这是两个容易混淆、但职责不同的功能：

| | AI 群聊分析（4.3） | 知识沉淀雷达（本节） |
|---|---|---|
| **职责** | **发现 + 结构化** | **沉淀 + 管理 + 流转** |
| **产出** | 报告 `.md` + 分析产物 `.artifact.json` | 知识库 `knowledge.db`（SQLite 卡片） |
| **候选是否入库** | ❌ 否，只是 artifact 里的候选文件 | ✅ 是，真正入库的卡片 |
| **可管理性** | 只能查看 / 删除文件 | 打分、状态流转、标签、转化、批量、导出 |

简单说：**AI 群聊分析是"发现器"，知识沉淀雷达是"沉淀器"。** AI 分析即使每天输出文件到 `ai_analysis/`，那些也只是"候选 + 报告"，并未进入知识库；真正入库（`knowledge.db`）、可被打分/收藏/转化/导出的知识沉淀，发生在这里。

两者通过"**知识来源**"打通（见下文）：知识雷达可以复用 AI 分析产出的候选，避免重复花 token。

#### 知识卡片字段

| 字段 | 说明 |
|------|------|
| 标题 / 类型 / 评分 | 卡片基本信息 |
| 摘要 / 为什么有价值 / 正文(Markdown) | 内容主体 |
| 标签 / 来源群 / 日期 | 分类与归属 |
| 来源(sources) | 每条引用的原始消息（群、发言人、时间、原文、所属话题、是否已验证） |

#### 卡片类型（9 种）

`审计案例 / SOP流程 / Prompt / FAQ / 文章 / 工具 / 风险 / 方法论 / 笔记`

#### 状态（Inbox 工作流）

| 状态 | 含义 |
|------|------|
| ⏳ 待沉淀 (`inbox`) | 新进入、待你审阅 |
| ⭐ 已收藏 (`saved`) | 确认有价值、长期保留 |
| 📦 已归档 (`archived`) | 暂不需要但留底 |
| 🚫 已忽略 (`rejected`) | 不需要的 |

状态可任意切换，无强制顺序。

#### 知识卡片从哪来：扫描方式 + 知识来源

点「开始扫描」前，先配两个关键选项：

**① 扫描方式**（如何切分时段，与 AI 分析的"分析方式"对应）：
- **按时间段扫描（默认）**：整个起止区间合并成一次扫描，产出的卡片日期标签为区间（如 `2026-06-01_to_2026-06-19`）。消息量大时会自动分块抽取再去重合并。
- **按天扫描**：区间内逐天各扫一次，卡片日期标签为单日。

**② 知识来源**（卡片从哪里生成，三档）：

| 来源 | 说明 | 适用场景 |
|------|------|---------|
| **高质量扫描原始消息（默认，`llm`）** | 直接读聊天文本，调 LLM 抽取并按多维度评分（可复用性 / 业务价值 / 稀缺性 / 结构化 / 证据价值 / 行动价值），达到阈值（默认 70 分）才入库 | 质量最高，不依赖 AI 分析是否跑过 |
| **优先复用 AI 分析结果（`auto`）** | 先查该群该时段**是否已有 AI 分析 artifact**，有就直接把候选转成卡片（省 token）；没有才回退到 `llm` 模式 | 已做过 AI 分析、想省钱 |
| **只使用已有 AI 分析结果（`artifact_only`）** | 只复用 artifact，没有就跳过，**不调 LLM** | 已批量分析过、只想快速入库 |

> **评分入库阈值**：可设最低分（默认 70），低于阈值的候选不入库。「按时间段扫描」消息量大时会自动分块，分块产出的重复卡片会按标题去重、合并来源与标签、保留最高分。

#### 另一条入库路径：从 AI 分析结果手动沉淀

除了上面的批量扫描，你也可以在「AI 群聊分析」结果页点「**沉淀高分知识**」，把单次分析的 artifact 中评分 ≥ 80 的候选**一键**导入知识库（相当于一次性的 `artifact_only`）。

找不到对应来源的候选会标注"证据缺失"由你决定，**不会**用首条消息凑数。

#### 可做的操作

- **筛选浏览**：按状态 / 类型 / 群标签 / 最低分 / 关键词过滤
- **单卡操作**：收藏 / 取消收藏、归档 / 取消归档、删除、编辑
- **批量操作**：全选当前列表、批量归档、批量删除、批量打标签
- **AI 转化**：把一张卡片用 AI 重写为更结构化的形式（如转成 SOP / Prompt / FAQ / 文章 / 审计案例 / 脚本）
- **溯源**：查看卡片引用的原始消息上下文
- **导出**：导出为 Markdown 或 Word（.docx），可按选中或当前筛选集导出；也可一键同步到 Obsidian vault（详见 [4.6](#46-同步到-obsidian-新)）
- **定时扫描**：配置每日自动扫描任务

### 4.6 同步到 Obsidian（新）

知识库存在 `knowledge.db`（SQLite 二进制），Obsidian 和 Claude Code 等 LLM 工具链默认无法直接读取。「同步到 Obsidian」把卡片导出成**一卡一文件**的 Markdown 笔记库，让这些工具能直接消费。

#### 为什么需要这个功能

| 方式 | 问题 |
|------|------|
| 直接用 Obsidian 打开 `knowledge.db` | Obsidian 只认 `.md` 文件，不读数据库 |
| 用 Claude Code 读 `knowledge.db` | 它读文本文件，二进制 SQLite 是乱码，需手动写 SQL |
| 现有「导出 MD」 | 把所有卡片拼成**一个大文件**，Obsidian 里不好用、无双链 |

同步到 Obsidian 后：每张卡片是独立笔记、按类型分目录、带 YAML 元数据、来源群名转 `[[双链]]`，Obsidian 的图谱/检索和 LLM 工具链都能直接用。

#### 操作步骤

1. 知识沉淀雷达页面 →「**⚙ 设置路径**」→ 输入 Obsidian vault 的本地绝对路径（如 `D:\Obsidian\MyVault`）
2. 点「**🔗 同步到 Obsidian**」→ 按当前列表筛选范围同步
3. 完成后显示「写入 N，跳过 N」
4. 用 Obsidian 打开该 vault 即可

同步范围复用知识库页当前的筛选条件（状态/类型/群标签/关键词/最低分）。

#### 产出的笔记长这样

按卡片类型分子目录，每张卡片一个文件：

```
MyVault/
├── SOP流程/
│   └── 某次盘点差异的排查SOP__a1b2c3d4.md
├── 审计案例/
├── FAQ/
├── Prompt/
└── ...
```

单个文件内容：

```markdown
---
id: a1b2c3d4-...
type: sop
score: 92
status: saved
date: 2026-06-15
tags: [盘点, SOP]
source_chats: [[审计交流群]]
updated_at: 1718400000
wechat_exp: "obsidian-export"
---

# 某次盘点差异的排查SOP

> 简介摘要……

## 为什么有价值
……

（正文 Markdown）

## 来源
- [[审计交流群]] 张三 (2026-06-15 10:32)：原文引用 ✓已验证
```

#### 增量同步（核心）

- 每次同步比对文件 frontmatter 的 `updated_at`，**相同就跳过**，只写真正变动的卡片
- 第二次同步通常很快（大多数卡片已存在且未变）
- 文件名带卡片 id 前缀防撞名（`标题__id8.md`）

#### 安全设计：只增不删

当卡片在知识库中被删除或移出筛选范围时，**Obsidian vault 里对应的旧文件不会被删除**。这是为了保护你在 Obsidian 里可能做过的手动编辑——宁可留冗余，不误删笔记。

#### 自动同步

除了手动点按钮，以下时机会自动同步（前提是已配置 vault 路径）：
- 知识雷达每次扫描入库后，自动顺带同步新增的卡片

#### 用 Claude Code / 其他 LLM 读取

同步后的 vault 就是一堆普通 `.md` 文件，Claude Code、Cursor 等可直接读取；frontmatter 的 YAML 元数据也能被工具解析用于检索、打标签、生成摘要。

## 6. AI 功能配置详解

AI 群聊分析、知识沉淀雷达、群自动分类都依赖一个 OpenAI 兼容的 LLM 服务。

### 6.1 配置入口

「AI 群聊分析」页面 →「⚙ LLM 设置」→ 填写并保存。配置一次后，三个 AI 功能共享同一套配置。

### 6.2 支持的 LLM 服务

任何兼容 OpenAI `/chat/completions` 接口的服务均可，例如：

- OpenAI 官方 API（`https://api.openai.com/v1`）
- DeepSeek、通义千问、Moonshot、智谱等国产兼容 API
- 自建 / 内网部署的 vLLM、Ollama（OpenAI 兼容模式）、One-API 网关等

### 6.3 配置项

| 配置 | 默认 | 范围 | 说明 |
|------|------|------|------|
| Base URL | — | — | API 根地址（以 `/v1` 结尾） |
| API Key | — | — | 界面掩码显示 |
| Model | — | — | 如 `gpt-4o` |
| Temperature | 0.3 | 0–2 | 越低越稳定 |
| Max Tokens | 4096 | 64–32000 | 单次响应上限 |
| Timeout | 120 秒 | 15–600 | 单次请求超时，超时自动重试 1 次 |

### 6.4 配置存储位置

- **主存储**：程序目录下的 `.wechat_exp_config.json` 的 `llm` 字段（UI 保存即写这里）
- **历史兼容**：`<备份根目录>/ai_analysis/config.json` 的 `llm` 字段作为 fallback，仅当主存储为空时读取

### 6.5 代理 / 科学上网

若访问 LLM 服务需要代理，程序已内置 **PySocks** 支持。设置系统环境变量即可（程序不提供代理输入框）：

```cmd
set HTTPS_PROXY=socks5://127.0.0.1:1080
set ALL_PROXY=socks5://127.0.0.1:1080
wechat_exp.exe serve
```

### 6.6 隐私提示

调用 AI 分析时，所选聊天记录的**文本内容**会发送给你配置的 LLM 服务。图片、语音、文件等媒体**不会**上传。请自行评估所用 LLM 服务的数据合规性。

## 7. V2 加密图片显示问题

### 7.1 什么是 V2 图片

微信 4.x 使用了新的图片加密方式（V2 加密），每张图有**唯一密钥**，只存在于微信运行时内存中，不落盘。

### 7.2 为什么有些图片无法显示

看到提示"**V2加密图片，暂不支持解码**"时，说明该图片密钥尚未被程序获取。

### 7.3 解决方法

1. 启动微信并登录
2. 在微信中**找到包含这些图片的聊天**
3. **滚动浏览这些图片**（让图片出现在微信界面上）
4. 仪表盘「工具」→「**提取密钥**」（或后台收割已自动运行）
5. 程序扫描微信内存，获取已加载图片的密钥
6. 刷新聊天查看器，图片即可正常显示

> **原理**：只有微信加载到内存中的图片，其密钥才能被提取，因此需要先在微信中浏览。备份前先浏览一遍有图片的重要聊天，可显著减少无法显示的情况。

## 8. 常见问题 (FAQ)

**Q1：双击 exe 后没有反应？**
可能被杀毒软件拦截。将 `wechat_exp.exe` 加入杀毒软件白名单，或以管理员身份运行。

**Q2：双击后弹出命令行窗口？**
正常现象。程序包含一个命令行服务窗口，会自动打开浏览器页面。**请勿关闭该窗口**，关闭即停止运行。

**Q3：浏览器没有自动打开？**
手动访问 `http://127.0.0.1:5051`。

**Q4：网页打开但显示"无法连接到后端"？**
检查命令行窗口未关闭、5051 端口未被占用。高级用户可通过命令行指定其他端口。

**Q5：备份时提示"未找到微信数据目录"？**
确认微信已安装且至少登录过一次（登录后会创建数据目录）。

**Q6：备份时提示"密钥不足"？**
微信只在登录后一小段时间内把所有数据库密钥加载到内存，时间久了老数据库密钥会被清除。解决方法（按推荐顺序）：
1. ⭐ **退出微信 → 重新登录 → 立即运行「一键备份」**
2. 确保微信正在运行且已登录
3. 仪表盘「工具」→「**提取密钥**」手动提取
4. 提取完成后重新备份

**Q7：图片显示"V2加密图片，暂不支持解码"？**
见 [第 7 章](#7-v2-加密图片显示问题)。

**Q8：AI 分析提示"LLM 未配置"？**
进入「AI 群聊分析」→「⚙ LLM 设置」配置 Base URL / API Key / Model 三项，测试连接通过后保存。

**Q9：AI 分析报"LLM 输出不是合法 JSON"？**
通常是模型能力不足或温度过高。尝试：换用更强的模型、把 Temperature 降到 0.1–0.3、适当提高 Max Tokens。

**Q10：AI 分析卡在超时？**
默认超时 120 秒，群多/消息长时可能不够。在 LLM 设置里把 Timeout 调高（最高 600 秒），或减少单次分析的群数量。

**Q11：程序会联网上传我的数据吗？**
聊天数据**完全在本地**，不会上传。但**调用 AI 功能时**，所选聊天记录的文本内容会发送给你配置的 LLM 服务——这是 AI 分析的必要环节，请在合规前提下使用。

**Q12：如何备份多个微信账号？**
程序自动检测第一个账号。切换时在微信登录目标账号 → 重新运行程序 → 备份。

**Q13：备份文件存在哪里？**
`wechat_exp.exe` 所在目录下的 `backup\` 文件夹，每次备份创建以日期命名的子文件夹。

### 9.6 典型工作流

```bash
# 首次使用：备份 → 收割密钥 → 启动查看
wechat_exp.exe backup --days 30
wechat_exp.exe harvest-keys
wechat_exp.exe serve

# 日常快速查看
wechat_exp.exe -q
wechat_exp.exe -q -c 张三

# 仅更新最近一周数据
wechat_exp.exe backup --days 7 --output "backup\latest"
wechat_exp.exe serve --decrypted-dir "backup\latest"

# 每日增量备份（复用前次媒体文件）
wechat_exp.exe backup --days 0 --wxid 你的wxid --link-dest "E:\wechat_bak\backup\2026-06-01"

# 导出综合报告
wechat_exp.exe export -m report
```

## 10. 附录：数据落盘位置汇总

| 数据 | 路径 | 说明 |
|------|------|------|
| LLM 配置 | `.wechat_exp_config.json` → `llm` | AI 功能共享 |
| 群黑名单 | `.wechat_exp_config.json` → `group_blacklist` | 全局剔除 |
| 数据库解密密钥 | `.wechat_exp_config.json` → `db_keys` | |
| Obsidian vault 路径 | `.wechat_exp_config.json` → `obsidian_vault_path` | 知识库同步目标 |
| 标签树 / 定时任务 | `<备份根>/ai_analysis/config.json` | 群管理与分析调度 |
| 群名缓存 | `<备份根>/ai_analysis/group_names_cache.json` | 7 天 TTL |
| AI 分析报告 | `<备份根>/ai_analysis/<群ID>/<日期标签>.md` | 人类可读 |
| AI 分析产物 | `<备份根>/ai_analysis/<群ID>/<日期标签>.artifact.json` | 结构化 JSON |
| 群风格指纹 | `<备份根>/ai_analysis/styles/<群ID>.json` | 仅群级 |
| 知识卡片库 | `<备份根>/ai_analysis/knowledge.db` | SQLite |
| Obsidian 笔记 | `<你配置的 vault 路径>/<类型目录>/<标题>__<id>.md` | 一卡一文件 |

> `<备份根>` 即备份数据目录（`backup\<日期>\<账号>\` 下的账号目录）。`<群ID>` 会把非字母数字字符替换为 `_`。`<日期标签>` 取决于分析方式：按天为 `2026-06-15`，按时间段为 `2026-06-01_to_2026-06-19`。
